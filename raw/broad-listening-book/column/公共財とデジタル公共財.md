## コラム：公共財とデジタル公共財

文責：tokoroten

### 「使うと減る」公共サービス

経済学では、財を「排除性」（対価を払わない人を排除できるか）と「競合性」（誰かが使うと他の人の分が減るか）の二つの軸で分類する。

|  | 非競合（使っても減らない） | 競合（使うと減る） |
|---|---|---|
| **非排除**（排除できない） | 純粋公共財（街灯、灯台、国防、一般道路、ラジオ放送） | コモンプール財（牧草地、漁場、地下水、水道、公園、図書館） |
| **排除可能** | クラブ財（有料放送、会員制ジム、高速道路、映画館） | 私的財（食べ物、衣服、自家用車） |

街灯や灯台、国防は純粋公共財の典型例だ。誰かが使っても他の人の分が減らず（非競合）、対価を払わない人を排除することもできない（非排除）。だからこそ税金で整備される。

コモンプール財には「コモンズの悲劇」という有名な問題がある。村で共有している牧草地で、各人が自分の利益を最大化しようと家畜を増やし続ける結果、牧草地全体が荒廃してしまう、という寓話だ。誰でも使えるが使うと減るものは、放っておくと枯渇する。経済学を超えて広く知られる概念であり、環境問題から行政サービスまで、あらゆる共有資源の問題を語るときに引き合いに出される。

もちろん、この4分類はきれいに分かれるわけではなく、現実にはグラデーションがある。重要なのは**利用者と提供者の比**だ。行政の窓口は1人の職員が1日に対応できる人数に上限がある。道路は同時利用者が増えれば渋滞する。行政サービスを含め、世の中の「公共的なもの」の多くは、提供者1人あたりが支えられる利用者数に限りがあるコモンプール財寄りの性質を持っている。

### コンピュータという労働者が提供する公共財

デジタルの世界では、この利用者/提供者の比が根本的に変わる。

ソフトウェアのソースコードは、何人がコピーしても元のコードは減らない。1人の開発者が書いたコードを100万人が使える。窓口職員1人が対応できる市民の数とは、桁が違う。そして、利用者が増えるほどバグが発見され、改善の機会も増える。つまりOSS（誰でもソースコードを見て、使って、改良できるソフトウェア）は、**提供者1人あたりが支えられる利用者数が極めて大きい、限りなく純粋公共財に近い財**なのだ。デジタルはコモンズの悲劇を乗り越えられる。

国連は2020年の「デジタル協力ロードマップ」[^un_roadmap]で、デジタル公共財を「オープンソースソフトウェア、オープンデータ、オープンAIモデル、オープンスタンダード、オープンコンテンツであって、プライバシーやその他の適用法令を遵守し、害を及ぼさず、SDGsの達成に貢献するもの」と定義した。この定義に基づき、Digital Public Goods Alliance（DPGA）[^dpga]が設立され、2025年時点で222件のデジタル公共財が登録されている。

この比率が大きいとはいえ、維持・改善する人の手はゼロではない。利用者が100万人に増えても、バグ報告やセキュリティ対応に応じる開発者が数人しかいなければ、対応は追いつかない。実際に、インターネットの基盤を支えるOSSが1〜2人のボランティアに依存していた事例は珍しくない。デジタル公共財は放っておけば勝手に良くなるのではなく、支える人と資金があってこそ良くなるのだ。

同じOSSを導入しても、ベンダーに丸投げする組織と、内部に技術を理解する人材がいる組織では、運用コストに桁違いの差が出る。デジタル公共財は無料だが、ゼロコストではない。使いこなすための人材こそが、デジタル公共財の真のコストだ。だからこそ政府は、道路や公園と同様にデジタル公共財の開発に投資するだけでなく、それを使いこなす人材にも投資する必要がある。

### 政府がOSSを「使う」から「作る」へ

各国政府もデジタル公共財の重要性を認識し、政策として推進し始めている。

米国では2016年、オバマ政権がFederal Source Code Policy（M-16-21）[^m1621]を発表した。連邦政府機関が新規に開発するカスタムコードの**少なくとも20%をオープンソースとして公開すること**を義務づけるものだ。また、ソフトウェア調達にあたっては「既存の政府ソリューションやOSSの採用→商用製品の購入→新規開発」という優先順位（Adopt, Buy, Create）を設定し、まずオープンなものから検討することを求めた。

この流れは軍にも及んでいる。2022年には国防総省CIOが「プロプライエタリ製品よりもオープンソースを優先採用する」方針を公式に打ち出した[^dod_oss]。安全保障上最も厳格な組織が、OSSを「より安全で経済的な選択」と位置づけたのだ。さらに、国防総省の職員がオープンソースプロジェクトに公務として貢献することを「政府資源の正当な使用」と明記した。

EUもデジタル主権の確保を掲げ、大手IT企業への依存を減らすためにオープンソース技術の活用を推進している。英国はGovernment Digital Service（GDS）の設立（2012年）以降、行政機関が約16,000件のOSSリポジトリを公開するまでに至った[^ipa_govoss]。

日本でも、IPAが2025年に公開したオープンソース推進レポート[^ipa_report]で、OSSを公共財として位置づける戦略の必要性を提言している。しかし同レポートは、日本の行政機関によるOSS公開数が英米仏に比べて大幅に少なく、企業もOSSを「使うだけ」の状態にとどまっていると指摘している。

IPAの行政OSS公開活動の国際比較調査[^ipa_govoss]では、英国の約16,000リポジトリに対し、日本は626件にとどまる。なぜ日本では進まないのか。その背景の一つに、地方自治法の壁がある。地方自治法第238条は、自治体が持つソフトウェアの著作権を「公有財産」と定めており、同法第238条の4は行政財産の譲与等を原則禁止している。このため自治体の法務担当者が「OSS化は公有財産の無償譲与にあたり違法ではないか」と懸念し、公開に踏み切れない状況が生じていた[^hal_dpg]。

### デジタル公共財を下敷きにした政党

デジタル公共財の概念を実際の政治に持ち込んだ事例がある。本書で繰り返し登場するチームみらいの代表、安野貴博は2025年5月の結党時にこう語った[^anno_100days]。

> 「普通、政党助成金というのは選挙活動に使われるわけですが、それを我々はデジタルの公共財を作っていくためのエンジニアチームに全力で使いたいと思っています」

政党交付金で10人規模のエンジニアチームを組織し、そのアウトプットを全てオープンソースで公開する。営利ではなく公共財の開発に専念する。AIの発展でスモールチームでも大きな成果が出せるようになったという時代認識のもと、永田町エンジニアチームがいどばたシステム、広聴AIによるパブリックコメント改革、自治体スーパーアプリなどを開発し、他党や他自治体にも技術を提供する、安野氏の言う「ユーティリティ政党」（使い勝手の良い政党）を目指すと宣言した。

実際にチームみらいは、政治資金を可視化するツール「みらい まる見え政治資金」をOSSとして公開した。2025年の参院選では、マニフェストをGitHub（ソフトウェアの設計図を公開・共同編集するプラットフォーム）上で公開し、市民からの改善提案を受け付けていた（6章参照）。

さらに安野議員は2025年の参議院総務委員会で、前節で触れた地方自治法の壁を正面から取り上げた[^anno_kokkai]。「公費で作ったコードは公共財として公開すべきだ」という主張に対し、林芳正総務大臣はOSS化は著作権の「利用許諾」であって著作権そのものの「譲与」ではないため、地方自治法上の制約はないと答弁した。総務省はこの解釈を全国の自治体に通知する方針を示している。

GitHubでのマニフェスト公開、OSSツールの開発、国会での質問。チームみらいの様々な活動を、デジタル公共財という概念で補助線を引くと「税金で作ったものは公共財として社会に還元するべきである」という考え方で一つに繋がって見えてくるのだ。

### 「予算」から「人材」の競争へ

地方自治法の壁が取り除かれ、デジタル公共財が充実すると、自治体間の競争の性質が変わる。同じOSSが無料で手に入る時代には、差別化の要因が「システムを買う予算」から「使いこなす人材」に移る。OSSをカスタマイズして住民サービスを素早く改善できる自治体と、ベンダーに丸投げする自治体とでは、サービスの質・コスト・スピードのすべてに差がつく。3章で述べた「足による投票」が、自治体内のデジタル人材の有無によって加速する未来だ。

この競争は悪いことではない。公共のOSSという共通の土台があるからこそ、自治体はゼロからシステムを作る必要がなくなり、住民サービスの改善に集中できる。競争の土台を公共財として整備し、その上で各自治体が独自の工夫を競う。デジタル公共財は、自治体に「マンパワーで解決する組織」から「デジタルの力で解決する組織」への転換を迫ることになるだろう。
